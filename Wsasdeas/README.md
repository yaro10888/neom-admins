# نظام إدارة استوديو ألعاب Roblox

لوحة تحكم متكاملة تعمل كنظام الإدارة الرسمي لاستوديو تطوير ألعاب Roblox:
إدارات، رتب، صلاحيات، حسابات، مطورون، مشاريع، مهام، مالية، دعاية، ملاحظات خاصة،
بيانات رسمية، وسجل نشاط كامل.

الواجهة عربية بالكامل (RTL) مع وضع ليلي افتراضي وتصميم متجاوب.

---

## التقنيات

| التقنية | الدور |
| --- | --- |
| Next.js 16 (App Router) | الواجهة، مُصدَّرة كملفات ثابتة |
| React 19 + TypeScript 5 | المكوّنات وأمان الأنواع |
| Tailwind CSS 4 | التنسيق |
| Supabase (Postgres + Auth) | قاعدة البيانات والحسابات والصلاحيات |
| Cloudflare Pages | الاستضافة |

المخططات البيانية مكتوبة بـ SVG خالص بدون مكتبات رسم.

---

## التشغيل محلياً

```bash
npm install
cp .env.example .env.local   # ثم ضع قيم مشروعك
npm run dev                  # http://localhost:3000
```

```bash
npm run build                # ينتج مجلد out/ الجاهز للرفع
npm run lint
```

المتطلبات: Node.js 20.9 أو أحدث.

## إعداد Supabase

1. أنشئ مشروعاً على [supabase.com](https://supabase.com).
2. **SQL Editor ← New query** ← الصق كامل [`supabase/schema.sql`](supabase/schema.sql) ← **Run**.
3. أضف رابط ويبهوك ديسكورد (يبقى داخل قاعدة البيانات ولا يصل للمتصفح أبداً):

   ```sql
   insert into public.app_secrets (key, value)
   values ('discord_webhook_url', 'https://discord.com/api/webhooks/...')
   on conflict (key) do update set value = excluded.value;
   ```

4. **Authentication ← Sign In / Providers ← Email** ← أوقف **Confirm email**.
   نظام تفعيل الحسابات في الموقع يغني عنه، وإبقاؤه مفعّلاً يمنع الدخول قبل تأكيد
   البريد ويصطدم بحدود الإرسال في Supabase.
5. انسخ من **Settings ← API Keys**:

   ```bash
   NEXT_PUBLIC_SUPABASE_URL=https://xxxxxxxx.supabase.co
   NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=sb_publishable_xxxxxxxx
   ```

> **لا تضع `sb_secret_...` في المشروع إطلاقاً.** هذا المفتاح يتجاوز كل الحماية،
> والمشروع لا يحتاجه. المفتاح العلني (`publishable`) آمن لأن الحماية الحقيقية
> في سياسات RLS داخل قاعدة البيانات.

## النشر على Cloudflare Pages

1. ارفع المشروع على GitHub.
2. Cloudflare ← **Workers & Pages ← Create ← Pages ← Connect to Git** ← اختر المستودع.
3. إعدادات البناء:

   | الحقل | القيمة |
   | --- | --- |
   | Framework preset | None |
   | Build command | `npm run build` |
   | Build output directory | `out` |

4. في **Settings ← Environment variables** أضف المتغيّرين:
   `NEXT_PUBLIC_SUPABASE_URL` و `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`.
5. **Save and Deploy**. كل دفعة جديدة على GitHub تُنشر تلقائياً.

المشروع مُصدَّر كملفات ثابتة (`output: 'export'`) فلا يحتاج خادماً ولا adapters —
أبسط وأثبت طريقة للنشر على Cloudflare.

## الحسابات والصلاحيات

**أول دخول:** يظهر زران — إنشاء حساب أو تسجيل الدخول.

- **إنشاء حساب:** الاسم، العمر، اسم Roblox، اسم ديسكورد، الإيميل، كلمة المرور.
  لا يُقبل تكرار أي من هذه البيانات مع حساب موجود.
- **تسجيل الدخول:** الاسم، اسم ديسكورد، الإيميل، كلمة المرور. يتم التحقق من
  الإيميل وكلمة المرور عبر Supabase Auth، ثم يُطابَق الاسم واسم ديسكورد مع الحساب.

**التفعيل:** كل حساب جديد يرى شاشة «حسابك بانتظار التفعيل» ولا يصل إلى أي بيانات
حتى تفعّله الإدارة من صفحة **الحسابات**، حيث تُسند له الإدارة والرتبة.

**حساب المالك:** أول حساب باسم `YARO` يصبح مالكاً مفعّلاً بكل الصلاحيات تلقائياً،
ولا يمكن إنشاؤه أكثر من مرة، ولا إيقافه ولا حذفه ولا تغيير رتبته (المنع مفروض في
قاعدة البيانات لا في الواجهة).

**الصلاحيات:** كل رتبة تحمل قائمة بصيغة `resource.action` مثل `projects.edit`،
والرمز `*` يمنح كل شيء. لوحة الصلاحيات في صفحة الرتب تُبنى تلقائياً من قائمة
الأقسام، فأي قسم جديد يظهر فيها فوراً.

> **الحماية الحقيقية في قاعدة البيانات.** إخفاء الأقسام في الواجهة لتحسين
> الاستخدام فقط؛ سياسات RLS في Postgres هي ما يمنع فعلياً أي محاولة للوصول
> المباشر إلى بيانات لا تسمح بها رتبة المستخدم.

## البيانات الرسمية

- **إرسال بيان** — يظهر فقط لمن تملك رتبته صلاحية `statements.create`.
  يحتوي على: العنوان، نص البيان، وتحديد ما إذا كان صادراً بأمر من رتبة أعلى،
  مع **معاينة الملف قبل الإرسال**.
- **البيان المرسل** — يظهر لمن يملك `statements.view`، ويعرض رقم البيان والتاريخ
  والحساب المرسِل مع زر لعرض معلوماته، وملف PDF يحتوي كل تفاصيل البيان والمرسِل
  (الإدارة، الرتبة، مستواها، التخصص، الحالة، ديسكورد، Roblox، الإيميل).

## النسخ الاحتياطي وإشعار ديسكورد

من صفحة **النسخ الاحتياطي**:

- **تحميل نسخة JSON** — نسخة كاملة قابلة للاستيراد لاحقاً.
- **تقرير PDF** — تقرير منظم بكل البيانات.
- **استيراد نسخة** — يتحقق من صحة الملف ويعرض محتواه قبل الاستبدال.

عند إنشاء أي نسخة (بأي صيغة) يُسجَّل صف في `backup_logs`، ومحفّز في قاعدة البيانات
يرسل إشعاراً إلى ديسكورد يحتوي: **اسم الاداري، اسمه في ديسكورد، التوقيت، النسخة**.

> **لماذا الطباعة بدل توليد PDF برمجياً؟** مكتبات PDF في المتصفح لا تصل حروف
> العربية ولا تعالج اتجاه النص إلا بخط مضمّن ومحرك تشكيل، والنتيجة حروف مقطّعة.
> المتصفح يعرض العربية سليمة دائماً، فمسار الطباعة يعطي ملفاً صحيحاً ومقروءاً.

## بنية المشروع

```
src/
├── app/                     # الصفحات (App Router)
│   ├── page.tsx             # الرئيسية
│   ├── departments/  ranks/  members/  members/view/
│   ├── developers/  projects/  tasks/  campaigns/
│   ├── finance/  notes/  activity/  users/
│   ├── statements/          # البيانات: الإرسال، المرسل، وملف PDF
│   ├── backup/              # النسخ الاحتياطي + تقرير PDF
│   └── settings/
│
├── core/                    # منطق العمل — لا يعرف شيئاً عن الواجهة
│   ├── domain/              # النماذج، الصلاحيات، الثوابت
│   ├── storage/             # المستودعات فوق Supabase
│   ├── services/            # حسابات مشتقّة: مالية، إشعارات، بحث، نسخ احتياطي
│   └── utils/               # التنسيق، المعرّفات، تحويل أسماء الحقول
│
├── components/
│   ├── ui/                  # مكوّنات معاد استخدامها
│   ├── layout/              # الهيكل وحارس الصلاحيات
│   ├── auth/                # شاشتا الدخول وانتظار التفعيل
│   └── features/            # مكوّنات مركّبة
│
├── providers/               # المخزن المركزي (React Context)
└── lib/supabase/            # عميل Supabase

supabase/schema.sql          # مخطط قاعدة البيانات الكامل
```

`core/` منطق خالص لا يستورد React أو Next، و`components/` عرض فقط،
و`providers/` هو الجسر بينهما.
